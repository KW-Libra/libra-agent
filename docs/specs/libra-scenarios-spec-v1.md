# LIBRA 멀티 에이전트 행동 시나리오 및 시스템 명세 v1

> **문서 목적**: LIBRA의 판단 에이전트 중심 동적 호출 구조를 구체적인 행동 시나리오로 검증하고, 그 과정에서 도출된 시스템 명세를 정리한다.
>
> **작성 배경**: 2026-04 교수 미팅에서 지적된 "왜 멀티 에이전트인가 / 왜 동적 호출인가 / 명세가 없다" 문제에 대한 응답으로 작성됨.
>
> **상태**: 1차 초안. 팀 리뷰 필요.

## 0. 핵심 주장 (Executive Summary)

LIBRA는 단일 종착지에서 결과를 종합하는 정적 파이프라인이 아니다. **판단 에이전트(Judge Agent)가 오케스트레이터로서 현재 상황에 따라 어떤 하위 에이전트를, 어떤 질문으로, 어떤 깊이로 호출할지 동적으로 결정**하는 ReAct 기반 멀티 에이전트 시스템이다.

이 동적 호출 구조의 정당화는 다음 세 가지에 근거한다:

1. **비용 효율**: 평범한 날엔 절반의 에이전트만 호출 (시나리오 C)
2. **추론 깊이**: 앞 에이전트의 답이 다음 에이전트의 질문 자체를 바꾼다 (시나리오 A)
3. **설명가능성**: 자연어 추론 흐름이 그대로 Decision Trace가 되어 사용자에게 "왜 그렇게 결정했는지"를 설명할 수 있다 (전 시나리오)

LIBRA의 핵심 가치는 alpha 생성이 아니다. **"직장인 다이렉트 인덱싱 투자자가 자기 투자 철학을 유지하면서도 매일의 리밸런싱 판단을 AI에게 위임할 수 있게 한다. 그 판단이 왜 내려졌는지 항상 설명 가능하다."** 수익률은 목표가 아니라 "기계적 리밸런싱 대비 유의미하게 떨어지지 않는다"는 제약 조건이다.

## 2. 시나리오 A — 정기 점검 + 신호 충돌 (DEFER 결말)

### 2.1 상황

- **시각**: 2026-04-30 목 08:50 (장 시작 10분 전)
- **트리거**: 정기 (Judge가 깨어남)
- **외부 이벤트**:
    - 전일 17:02: 삼성전자 1Q 잠정실적 공시 (영업이익 컨센서스 -12%, 회사는 "메모리 가격 하락" 명시)
    - 야간: 미국 마이크론 어닝 서프라이즈, 시간외 +6%
    - 증권사 코멘트 리포트: 미발간 (오전 중 예정)

### 2.2 추론 흐름

#### 턴 1 — Disclosure Agent 스캔

Judge의 첫 질문은 "삼성전자를 어떻게 할까"가 아니라 **"내가 지금 알아야 할 게 뭐가 있나"**다. 비중 편차만 보면 트리거 안 걸리지만, 야간 이벤트가 오픈 가격에 반영될 수 있으므로 사전 스캔.

```
call_disclosure_agent(
  query: "포트폴리오 4종목 어제 15:30 이후 신규 공시",
  priority: "정기 사전 스캔"
)
```

**Disclosure Agent 응답**: 삼성전자 잠정실적 공시 1건 발견. 매출 부합, 영업이익 -12% 미스, 회사 자체 코멘트 "메모리 가격 하락".

> **중요: 디자인 포인트**
> Disclosure Agent는 점수를 매기지 않는다. `raw_signal: "어닝 미스"`라는 정성적 라벨만 준다. 점수화는 Judge가 모든 정보를 모은 다음에 한다. 이게 ReAct 정신이다 — 하위 에이전트가 너무 빨리 단정하면 Judge의 판단 공간이 좁아진다.

#### 턴 2 — News 호출 (구조적 vs 일시적 판별)

Judge의 추론: "회사가 직접 메모리 가격이라고 했으면 산업 전체 이슈일 가능성. 그러면 마이크론 결과가 결정적이다. 산업 사이클 바닥이면 매수 기회, 약세 지속이면 비중 축소."

```
call_news_agent(
  query: "야간 미국 메모리 섹터 동향, 마이크론 실적과 가이던스, 한국 매체 반응",
  context: "삼성 어닝 미스가 산업 구조적 이슈인지 일시적 노이즈인지 가려야 함",
  depth: "deep"
)
```

> **중요: 디자인 포인트**
> Judge가 News에 던지는 호출에 `context` 필드가 들어간다. 단순 검색 위임이 아니라 **"이런 맥락에서 묻는다"는 의도 전달**. 이래야 News 에이전트가 자기 검색 깊이와 분석 방향을 조정할 수 있다.

**News 응답**: MIXED. 마이크론 자체는 호실적이지만 가이던스는 "DRAM 안정화 / NAND 약세"로 갈림. 섹터 ETF 반응은 마이크론 개별 호재로 해석 중. 한국 매체 톤도 갈림. **confidence 0.55**.

News는 빈손이 아닌 풍부한 답을 주면서, `reasoning_for_judge_agent` 필드로 **"리포트 에이전트나 추가 정보 필요"**라고 다음 행동까지 제안.

#### 턴 3 — Report 호출 (사업부별 분해)

Judge의 추론: "마이크론 가이던스가 DRAM/NAND 갈림. 삼성 미스가 어느 쪽인지에 따라 해석 정반대. 리포트가 사업부별 분해를 줄 수 있는지 확인."

```
call_report_agent(
  query: "삼성전자 1Q 잠정실적 코멘트 리포트, 사업부별 미스 원인",
  context: "DRAM이냐 NAND냐에 따라 결론이 정반대",
  fallback: "코멘트 리포트 없으면 전일 프리뷰 리포트라도"
)
```

> **중요: 디자인 포인트**
> Judge가 호출에 `fallback` 옵션을 같이 준다. "원하는 정보가 없을 수 있다"는 가능성을 미리 인지하고 차선책까지 던지는 것. 이것도 ReAct의 일부.

**Report 응답**: DIRECT_ANSWER_UNAVAILABLE. 코멘트 리포트 0건. 프리뷰 리포트 3건만 있고, 그중 가장 보수적이었던 추정치도 실제보다 높았음 → **시장 전체가 미스 폭을 과소평가했다**는 간접 단서. 직접 증거 아님. **confidence 0.40**.

`reasoning_for_judge_agent`: "1시간 후 코멘트 리포트 발간 가능성 높음. '지금 결정' vs '대기'의 트레이드오프를 비용 에이전트한테 평가시켜라."

#### 턴 4 — Cost 호출 (확장 역할: "대기의 기회비용")

> **🎯 핵심: 가장 강력한 셀링 포인트**
> 이 턴이 LIBRA의 정체성을 가장 잘 보여준다. Judge가 비용 에이전트의 **정의된 역할(거래비용 계산)을 넘어서** "대기의 기회비용"이라는 새로운 질문을 던진다. 정적 파이프라인이라면 비용 에이전트는 항상 같은 종류의 질문만 받는다. 동적 호출 구조에서만 가능한 행동이다.

```
call_cost_agent(
  query: "시나리오 X(즉시 결정) vs Y(1시간 대기 후 결정) 비교 평가",
  context: "정보 부족 상황에서 결정 타이밍 자체를 따지는 중",
  note: "일반적 거래비용 계산이 아닌 '대기의 기회비용' 평가. 못 하면 못 한다고 답해도 됨"
)
```

**Cost 응답**: 과거 어닝 쇼크 7건 분석. 1시간 대기 시 평균 -2.8% / 최악 -6.4% 변동, 사용자 손실 한계(-5%) 위협 확률 5% 미만. 거래비용은 9시 직후(8bp) vs 10시 전후(4bp). 순수 비용/리스크는 대기 우세.

단, **"정보 quality 향상이 실제로 더 나은 결정으로 이어진다는 보장은 내가 평가 못 함"**이라고 자기 영역의 한계 명시. **confidence 0.62**.

#### 턴 5 — Judge의 최종 결정: DEFER

Judge의 추론 종합:
- 사용자 손실 한계 위협 < 5%
- 거래비용은 대기가 4bp 유리
- 정보 quality 향상 가능 (코멘트 리포트)
- 신호 충돌 상태에서 즉시 결정 시 "잘못된 방향 리밸런싱" 위험

→ **DEFER**. 09:55에 다시 깨어나서 Report 재호출 후 재판단.

사용자 알림 발송 (urgency: info): "어닝 미스 인지함, 신호 충돌 중, 1시간 후 재판단 예정, 사용자 액션 불필요".

### 2.3 시나리오 A의 의미

> **💡 결론이 "리밸런싱 안 함"이라는 게 핵심이다.**
> 보통 시연 시나리오는 "AI가 멋지게 사고팔았다"로 끝낸다. LIBRA는 정반대로 간다 — **"AI가 정보 부족을 인지하고 의사결정을 미뤘다, 그리고 그 이유를 사용자에게 자연어로 설명했다"**가 결말이다. 이게 사용자 편의성 + 책임 위임이라는 핵심 가치와 정확히 맞물린다. 기존 로보어드바이저는 매일 뭔가 하려고 들지만, LIBRA는 "할 필요가 없거나 할 정보가 부족하면 안 한다"고 사용자한테 설명하는 시스템이다.

## 3. 시나리오 B — 장중 속보 (USER_DECISION_REQUIRED 결말)

### 3.1 상황

- **시각**: 2026-05-14 수 14:17 (장중)
- **트리거**: 이벤트 push (News 에이전트가 Judge를 깨움)
- **외부 이벤트**: 14:15 연합뉴스 속보 — "현대차, 미국 NHTSA로부터 EV 배터리 화재 관련 조사 착수 통보"
- **사전 상태**: 오전 정기 점검에서 HOLD 결정 후 Judge는 잠들어 있던 상태

### 3.2 추론 흐름

#### 턴 0 — News의 자체 임계치 판단 ("이거 깨울 만한가?")

> **중요: 시나리오 A와의 결정적 차이**
> 시나리오 A에서 호출자는 Judge였다(pull). 시나리오 B에서 호출자는 News다(push). **호출 방향이 뒤집힌다.** 4월 명세에 명시 필요.

News가 깨우기 전 자체 검증:
1. 사용자 종목 관련성? → 예 (현대차)
2. 매체 cross-check? → 4개 매체, NHTSA 공식 인용 → 신뢰도 high
3. 가격 이미 반응? → 14:10~14:17 사이 -3.4%, 거래량 6.8x 급증
4. NHTSA 조사 단계? → Preliminary Evaluation (1/4단계). 과거 PE → 리콜 비율 30%

판단: 깨울 가치 충분. 단, **단순 알림이 아니라 사전 스크리닝 결과를 같이 전달**하여 Judge가 처음부터 컨텍스트를 갖고 시작하게 함.

#### 턴 1 — Judge 깨어남, 1차 인지 알림 즉시 발송

Judge의 추론: "장중이고 사용자가 앱을 열어볼 가능성 높음. 다른 매체에서 같은 뉴스를 먼저 보면 LIBRA가 늦게 반응했다는 인상을 줌. 분석 끝낸 뒤 알리지 말고, 먼저 '인지함, 분석 중' 알림을 보낸다. 내 분석에는 5분 deadline 건다."

```
send_user_notification(
  type: "event_acknowledged",
  body: "현대차 NHTSA 조사 보도 감지. 5분 이내 분석 결과 알려드림. 액션 불필요.",
  urgency: "watch",
  estimated_followup: "14:22"
)
```

> **중요: 디자인 포인트**
> Judge가 자기 자신에게 deadline을 부여한다. 5분 안에 결정 못 내리면 그 자체가 의미 있는 정보(수동 결정 요청 상태로 전환).

#### 턴 2 — Cost 우선 호출 (시나리오 A와 다른 호출 순서)

Judge의 추론: "이번엔 정보 수집보다 '이미 빠진 가격에 추가로 팔 가치가 있는가'가 핵심 질문. Cost부터."

Cost 응답: 절반 매도(X2)가 비용 효율 최적 (총 friction 23bp). 과거 유사 이벤트 11건 분석 시 발표 당일 평균 -4.1% 추가 하락, 30일 기준 +0.6% 회복. **"매도해야 하는가" 자체는 사용자 신념 영역, 내가 답할 수 없음.**

#### 턴 3 — Judge의 메타 결정: 자율성 경계 인식

> **🎯 핵심: 시나리오 B의 핵심 통찰**
> Cost가 "사용자 신념 확인 필요"라고 다시 던졌다. Judge는 사용자 사전 설정을 재확인하지만 **"규제 리스크 발생 시 어떻게 행동할 것인가"에 대한 명시적 지침이 없다**는 걸 발견한다.
>
> 여기서 LIBRA의 자율성 경계가 정의된다 — **사용자가 사전 위임한 영역 안에서만 자동 행동, 그 너머는 사용자에게 결정권 반납.** 이게 "판단 위임 + 설명가능성" 가치의 구체적 구현이다.

Judge의 결정: 권고안 제시 + 사용자 승인 요청 모드 전환. 단, 안전망(트립와이어)은 자동 설정.

#### 턴 4 — Cost 재호출 (트립와이어 가격 계산)

```
call_cost_agent(
  query: "현대차 트립와이어 가격 설정. -5% / -8% 두 단계",
  urgency: "5분 내"
)
```

응답: 178,125원 (절반 자동 매도) / 172,500원 (추가 절반 매도, 총 75% 청산).

#### 턴 5 — 사용자 결정 요청 통지 발송

```
{
  decision: "USER_DECISION_REQUIRED",
  recommendation: "현대차 비중 절반 축소 (20% → 10%)",
  options: ["권고안 승인", "전량 매도", "유지", "직접 비율 설정"],
  auto_safeguards: {
    tripwire_1: "178,125원 → 절반 자동 매도",
    tripwire_2: "172,500원 → 추가 절반 자동 매도"
  },
  total_elapsed: "247초 (deadline 내)"
}
```

### 3.3 시나리오 B의 의미

시나리오 A의 결말은 "DEFER (정보 부족 → 대기)"였다. 시나리오 B의 결말은 "USER_DECISION_REQUIRED (사전 위임 범위 밖 → 권한 반납)"다. 이 두 결말이 다르다는 것이 LIBRA의 정체성을 정의한다 — **자율적으로 모든 걸 처리하는 시스템이 아니라, 자율성의 경계를 인지하고 그 경계 안에서만 행동하는 시스템**.

## 4. 시나리오 C — 평범한 날 (HOLD silent 결말)

### 4.1 상황

- **시각**: 2026-05-21 수 08:50
- **트리거**: 정기
- **외부 이벤트**: 없음

### 4.2 추론 흐름 (3턴, 압축)

#### 턴 1 — Disclosure Agent 스캔
신규 공시 0건. 단, **"5/23 KB금융 1Q 실적 발표 예정"이라는 미래 이벤트를 함께 알림** (빈손 리턴 금지 원칙).

#### 턴 2 — News shallow 스캔

```
call_news_agent(
  query: "오늘 광범위 매크로/섹터 컨텍스트 점검",
  depth: "shallow"
)
```

> **중요: 디자인 포인트**
> `depth: "shallow"`. 시나리오 A의 `"deep"`과 대비. 같은 에이전트지만 호출 깊이가 다르면 도구 호출 횟수와 비용이 다르다.

응답: QUIET_DAY. 도구 2개만 호출 (deep 모드는 4~6개). confidence 0.85. "추가 분석 호출은 비용 낭비, HOLD 권고. 5/22 점검에서 KB 프리뷰 확인 필요."

#### 턴 3 — Judge: HOLD + 미래 점검 항목 등록

Report와 Cost는 호출하지 않음 (`agents_skipped`에 명시 + skip 이유 기록).

```
{
  decision: "HOLD",
  daily_summary: { urgency: "silent", body: "특이사항 없음, 다음 점검 5/22" },
  scheduled_followup: { date: "2026-05-22", checklist: ["KB금융 1Q 프리뷰 리포트"] },
agents_called: ["Disclosure", "News"],
  agents_skipped: ["Report", "Cost"],
  skip_rationale: "정보 수집 단계에서 특이사항 0건"
}
```

### 4.3 시나리오 C의 의미

평범한 날 시나리오의 가치는 두 가지:

1. **"안 부르는 것"도 의사결정이다.** 정적 파이프라인이면 매번 4개 다 부르지만, 동적 호출은 평범한 날엔 절반만 쓴다. LLM 호출 비용 관점에서 정당화의 근거.
2. **silent 알림이 사용자 경험의 핵심.** 평범한 날까지 푸시를 보내면 사용자는 LIBRA를 끈다. 알림 강도 4단계 분류(silent / info / watch / push)는 사용자 편의성 가치의 구체적 구현이다.
3. **메모리/피드백 루프의 첫 구현**: "5/22에 KB 프리뷰 봐라"라는 미래 점검 항목을 자기 자신에게 남긴다.

## 4.5. 시나리오 D — 자동 REBALANCE (실행 단계 정립)

### 4.5.1 상황
- **시각**: 2026-05-23 금 08:50 정기
- **외부 이벤트**: 전일 장 마감 후 KB금융 1Q 순이익 컨센 +18% 상회, 야간 KBE +2.3%, 목표주가 상향 리포트 예상
- **메모리 연결**: 시나리오 C에서 "5/22 KB 프리뷰 확인" 예약이 가동되어 선행 확인 완료된 상태
- **사용자 사전 설정**: "종목별 ±5% 내 자동 리밸런싱 허용" → 15%→20%는 자동 위임 범위 안

### 4.5.2 추론 흐름 (5턴 + 사후)

**턴 1 — Disclosure Agent + 메모리 로드**
Judge가 어제의 예약 점검 항목을 불러옴. Disclosure 공시 확인, KB금융 어닝 서프라이즈 raw_signal 수령. confidence 0.88.

**턴 2 — Report deep 호출**
목표주가 상향 3건 수령. 공통 코멘트: "충당금 환입은 일회성, NIM 개선은 2~3분기 지속". direction +0.62, strength 0.71, urgency scheduled, confidence 0.81. references: Disclosure(depend).

**턴 3 — News shallow 호출**
섹터 ETF 반응 + 한국 매체 톤 확인. direction +0.55, confidence 0.76. references: Report(agree).

**턴 4 — Cost 실행 계획 수립 (확장 역할)**
TWAP 5회 분할 (09:10/09:40/10:20/11:00/13:30), 예상 friction 20bp, abort 조건 -3%. direction 중립(비용 평가자). confidence 0.79.

**턴 5 — Judge 합의 + 자동 실행**
가중평균 direction +0.46, 분산 낮음 → CONSENSUS. 사전 위임 범위 안 → REBALANCE 자동 실행. 사용자 알림은 info 강도.

**턴 6 — 사후 알림**
13:40 실행 완료, 실제 슬리피지 9bp(예상 12bp 하회). feedback_checkpoint 2026-06-23 예약.

> **중요: 시나리오 D의 구조적 도전**
> 시나리오 A/B/C는 "안 하는 것의 정당화"였다면, D는 "하는 것의 정당화". 그리고 LIBRA 컴포넌트 중 **3번 리밸런싱 처리 기능**이 처음으로 구체적으로 다뤄지는 시나리오. Judge ↔ 실행 컴포넌트 인터페이스(work item E-2)의 실제 예시.

### 4.5.3 시나리오 D의 의미
1. REBALANCE도 설명 가능한 결정이다 — 매매해도 사용자가 "왜 했는지" 자연어로 이해 가능
2. 확장 역할(명세 5번)이 positive 방향으로도 작동 — Cost가 "실행 계획 수립"이라는 새 역할 수행
3. feedback_checkpoint 필드로 메모리/피드백 루프의 두 번째 구현체 진입
4. 시나리오 C에서 예약한 점검 항목이 D에서 가동되는 **메모리 연결**의 첫 증거

### 6.8 실행 + 피드백 명세 (15 ~ 17)

> **✓ 명세 15. 실행 단계 분리**

> **✓ 명세 17. Feedback Checkpoint**

## 6. 시스템 명세 (시나리오에서 도출된 14개 항목)

### 6.1 에이전트 응답 형식 명세

> **✓ 명세 1. Confidence 자체 보고**
> 모든 에이전트는 답변과 함께 `confidence` 점수(0.0~1.0)를 자체 보고해야 한다. confidence 산정 방법은 에이전트별로 정의된다.

> **✓ 명세 2. Reasoning 다음 행동 제안**
> 모든 에이전트는 답변에 `reasoning_for_judge_agent` 필드로 다음 행동 제안을 포함해야 한다. Judge는 이 제안을 받아들일 수도, 무시할 수도 있다.

> **✓ 명세 3. 빈손 리턴 금지**
> 정보가 없을 때도 의미 있는 답을 한다. 직접 답을 못 하면 간접 단서, 차선책, 미래 이벤트 알림 등을 제공한다.

> **✓ 명세 4. 자기 영역 한계 명시**
> 에이전트는 자기 영역의 한계를 명시하고, 그 너머의 판단은 Judge에게 위임한다.

### 6.2 Judge의 호출 권한 명세

> **✓ 명세 5. 확장 역할 호출 허용**
> Judge는 에이전트의 정의된 기본 역할을 넘어서는 확장된 질문을 던질 수 있다. 단, 호출 시 `note` 필드로 확장 의도를 명시한다. (예: 시나리오 A의 "대기의 기회비용" 호출)

> **✓ 명세 6. Context 필드**
> Judge가 하위 에이전트를 호출할 때는 `query`만이 아니라 `context`(이 질문을 던지는 맥락)를 함께 전달한다. 하위 에이전트는 context를 보고 자기 행동을 조정한다.

> **✓ 명세 7. Depth 파라미터**
> 모든 에이전트 호출은 `depth` 파라미터를 받는다 (`shallow` / `medium` / `deep`). 에이전트는 depth에 따라 도구 호출 횟수와 분석 깊이를 조정한다.

### 6.3 의사결정 결과 명세

> **✓ 명세 8. Decision Trace 자연어 기록**
> 의사결정 결과는 `decision_trace`에 턴 단위로 기록되어야 한다. 각 턴에는 호출된 에이전트, 던진 질문, 받은 답의 핵심이 자연어로 들어간다. Decision Trace는 사용자에게 노출 가능한 형태여야 한다.

> **✓ 명세 9. DEFER는 정상 결정 타입**
> "DEFER"(대기)는 HOLD/REBALANCE와 동급의 정상적인 의사결정 타입이다. DEFER 결정 시 재판단 시점과 사유가 명시되어야 하며, 사용자에게 통지된다.

### 6.4 트리거 및 호출 방향 명세

> **✓ 명세 10. 이벤트 트리거 (Push)**
> 뉴스 에이전트는 백그라운드 모니터링 중 임계치를 넘는 이벤트 감지 시 Judge를 깨울 수 있다(push). 이때 단순 알림이 아니라 사전 스크리닝 결과(매체 cross-check, 가격 반응, 영향 추정)를 함께 전달한다.

> **✓ 명세 11. Self-imposed Deadline**
> 이벤트 트리거 의사결정에는 self-imposed deadline이 적용된다. Deadline 초과 시 Judge는 "수동 결정 요청" 상태로 전환한다.

### 6.5 자율성 경계 명세

> **✓ 명세 12. 자율성 경계 — 사전 위임 범위 외 권한 반납**
> Judge는 사용자 사전 설정에 명시되지 않은 의사결정 카테고리를 마주하면, 자동 행동 대신 "권고안 + 승인 요청" 모드로 전환한다. (USER_DECISION_REQUIRED 결정 타입)

> **✓ 명세 13. 트립와이어 자동 작동**
> 사용자 승인 대기 중에도 손실 한계(하드 가드레일) 트립와이어는 자동 작동한다. 트립와이어는 단계적(예: -5% / -8%)으로 설정 가능하다.

### 6.6 사용자 알림 강도 명세

> **✓ 명세 14. 알림 4단계 강도 분류**
> 사용자 알림은 4단계 강도로 분류된다:
> - **silent**: 앱 내 기록만, 푸시 없음 (평범한 날 일별 요약)
> - **info**: 앱 내 + 비푸시 마크 (DEFER 등 인지 필요)
> - **watch**: 푸시 알림, 액션 불필요 (이벤트 인지 1차 알림)
> - **push**: 푸시 + 사용자 액션 요구 (USER_DECISION_REQUIRED)

## 6.7 협업 구조 + 시각화 명세 (C-1 ~ C-7)

> **💡 설계 방법론 노트**
> 다음 7개 명세는 일반적인 시스템 설계 순서를 거꾸로 거쳐 도출되었다. 먼저 사용자에게 보여줄 시각화 형식(타임라인 + 합의 게이지)을 정하고, 그 시각화가 작동하기 위해 시스템이 무엇을 보장해야 하는지를 역산했다. **사용자 경험 → 시스템 명세** 방향의 역설계는 4월 미팅에서 "사용자 편의성에 focusing 두라"는 교수님 권고에 직접 응답하는 작업이다.

> **✓ 명세 C-1. direction × strength × urgency 필수 필드**
> 모든 에이전트 응답은 `direction`(-1.0~+1.0, 매도~매수), `strength`(0~1), `urgency`(immediate/scheduled/watch/defer)를 명시한다. 합의 게이지의 두 축(방향 × 긴급도) 시각화의 기반.

> **✓ 명세 C-2. 다른 에이전트 의견 참조 권한**
> 에이전트는 응답에 `references: [{agent_id, opinion_id, relation}]` 필드를 포함할 수 있다. relation은 `request` / `rebut` / `agree` / `depend` 4종. 이를 통해 토론 타임라인의 화살표 의미 분류가 가능해진다.

> **✓ 명세 C-3. Turn + Phase 마커**
> Judge는 매 의사결정 턴마다 `turn_number`와 `phase`를 명시 (`information_gathering` / `deliberation` / `consensus` / `decision`). 시간 축이 "초"가 아니라 "추론 단계"가 되어 사용자에게 자연스럽게 전달됨.

> **✓ 명세 C-4. 도구 호출 사용자 노출 로깅**
> 모든 도구 호출은 자연어 요약 형태로 기록된다. 카드 펼침 시 사용자에게 "어떤 정보 출처를 사용했는지" 노출 가능해야 함.

> **✓ 명세 C-5. 명시적 합의 형성 알고리즘**
> Judge의 합의 도출은 가중평균(direction을 confidence로 가중) + 분산 기반 DEFER 트리거. 분산이 임계치 초과 시 자동으로 DEFER 또는 USER_DECISION_REQUIRED. 알고리즘은 결정론적이고 사용자에게 설명 가능해야 함.

> **✓ 명세 C-6. Judge 경유 통신 + 참조 권한 (B + C 하이브리드)**
> 모든 에이전트 호출은 형식적으로 Judge를 경유한다(B 구조). 단 응답에서 다른 에이전트의 의견을 참조하고 반박할 수 있다(C 정신). 완전한 P2P 토론은 구현하지 않으며, 이 절충은 시각화 요구사항에서 자연스럽게 도출된 결과다.

> **✓ 명세 C-7. Decision Trace는 노드 리스트 + 참조 필드**
> Trace는 그래프가 아닌 "시간순 노드 리스트 + 각 노드의 references 필드"로 저장. 시각화 렌더링 시 그래프로 펼칠 수 있고, 자연어 설명도 선형으로 가능.

## 7. 시나리오에서 발견된 진짜 어려운 문제 (4월 설계 과제)

다음은 시나리오를 짜는 과정에서 "어떻게 풀지?"가 명확하지 않았던 항목들이다. 4월 설계의 핵심 work item이 된다.

1. **각 에이전트의 confidence 점수를 어떻게 계산할 것인가** — 정성적 자체 보고로 갈지, 정량 모델을 둘지, 결정 필요. 가장 어려운 문제.
2. **Judge가 언제 "정보 충분"이라고 결정할 것인가** — 시나리오 A에서 자연스럽게 풀렸지만 일반화 알고리즘 필요.
3. **DEFER 결정의 트리거 시간을 어떻게 자동 결정할 것인가** — "1시간 후"는 어떻게 도출되는가?
4. **에이전트의 확장 역할 호출을 어떻게 안전하게 허용할 것인가** — Judge가 무한히 확장하면 비용 폭발 + 신뢰도 하락. 가드레일 필요.
5. **News 에이전트의 push 임계치를 어떻게 정의할 것인가** — false positive 비용 vs false negative 비용 트레이드오프.
6. **사용자 사전 설정의 빈틈을 어떻게 식별하고 사용자에게 빈틈 채우라고 요청할 것인가** — 시나리오 B에서 드러난 문제.
7. **트립와이어 설정 권한을 Judge에게 어디까지 줄 것인가** — 자동 매도 가능한 영역의 명시적 정의.

이 항목들은 LIBRA 4월 Work Items에 작업 단위로 분해되어 있다.
