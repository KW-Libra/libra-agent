# LIBRA 데모 스크립트

## 데모 목적

교수님에게 LIBRA가 단순 화면 목업이 아니라, 목표 포트폴리오 설정부터 AI 판단, Decision Trace, KIS 현재가 기반 주문 제안까지 연결되는 시스템임을 보여준다.

## 데모 전제

- 실행 환경: 로컬 개발 환경
- Backend: `http://127.0.0.1:8080`
- Agent: `http://127.0.0.1:8010`
- Frontend: `http://127.0.0.1:5173`
- KIS 환경: 모의투자 기준

## 시연 순서

1. 로그인한다.
2. 초기 설정 또는 포트폴리오 설정 화면에서 종목과 목표비중을 확인한다.
3. 예시 포트폴리오를 설명한다.
   - 삼성전자 50%
   - SK하이닉스 50%
4. Dashboard에서 현재 포트폴리오 상태를 확인한다.
5. KIS 잔고 동기화가 가능한 경우 모의투자 잔고를 동기화한다.
6. 판단 실행 버튼을 눌러 Judge Run을 생성한다.
7. History 화면으로 이동한다.
8. 방금 생성된 판단 이력을 연다.
9. 최종 판단과 후보 리밸런싱 초안을 확인한다.
10. Decision Trace에서 Judge가 어떤 에이전트를 호출했는지 보여준다.
11. `KIS 현재가로 수량 산출` 버튼을 누른다.
12. 종목별 매수/매도 수량, 가격, 금액을 확인한다.
13. 모의투자 주문 실행 게이트가 있다는 점을 보여준다.

## 설명 멘트

“사용자가 초기에 종목과 목표비중을 설정하면, 시스템은 현재 포트폴리오와 목표 포트폴리오의 차이를 계산합니다. 그 차이가 리밸런싱 임계치를 넘으면 후보 리밸런싱 초안이 생성되고, Judge Agent가 공시·뉴스·수익·비용 관점의 에이전트 의견을 종합해서 최종 판단을 내립니다. 판단 결과는 Decision Trace로 남고, REBALANCE일 경우 KIS 현재가를 기준으로 실제 주문 수량까지 계산합니다.”

## 데모 실패 시 대체 설명

- KIS 호출이 실패하면: “외부 API 실패는 현재가 산출 단계의 문제이고, Judge 판단과 Decision Trace 저장은 이미 분리되어 동작합니다.”
- Agent 호출이 느리면: “Claude API 호출이 포함되어 있어 응답 시간이 걸릴 수 있으며, 로컬 테스트에서는 동일 입력으로 REBALANCE와 주문 제안 생성까지 확인했습니다.”
- 모의주문 실행을 못 하면: “현재는 실전 자동매매가 아니라 사용자 승인 게이트가 있는 모의투자 실행 경로까지를 범위로 잡았습니다.”
