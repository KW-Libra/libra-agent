# LIBRA 요구사항 명세 초안

## 서비스 정의

LIBRA는 개인 투자자가 선택한 종목과 목표비중을 기준으로 포트폴리오를 구성하고, 주기적으로 현재 포트폴리오 상태와 시장 정보를 확인하여 리밸런싱 여부를 판단한 뒤, 판단 근거와 실행 후보를 사용자에게 제공하는 웹 기반 시스템이다.

## 기능 요구사항

1. 사용자는 이메일 기반 회원가입과 로그인을 할 수 있어야 한다.
2. 사용자는 한국투자증권 모의투자 또는 실전투자 API 자격증명을 등록할 수 있어야 한다.
3. 시스템은 KIS 종목 정보를 조회하여 사용자가 초기 포트폴리오 종목을 선택할 수 있게 해야 한다.
4. 사용자는 선택한 종목별 목표비중을 설정할 수 있어야 하며, 목표비중 합은 100%가 되어야 한다.
5. 시스템은 사용자의 목표 포트폴리오 정의를 저장하고 이후 판단 실행에 자동으로 사용해야 한다.
6. 시스템은 KIS 잔고 동기화를 통해 현재 보유 종목, 수량, 평가금액, 현재비중을 가져올 수 있어야 한다.
7. 시스템은 목표비중과 현재비중의 차이를 계산하여 종목별 드리프트를 산출해야 한다.
8. 시스템은 드리프트가 임계치를 넘을 경우 후보 리밸런싱 초안을 생성해야 한다.
9. 시스템은 공시, 뉴스, 리포트, 수익, 비용 관점의 에이전트 응답을 공통 형식으로 정규화해야 한다.
10. Judge Agent는 상황에 따라 필요한 하위 에이전트를 선택적으로 호출해야 한다.
11. Judge Agent는 호출한 에이전트와 호출하지 않은 에이전트의 이유를 decision trace에 기록해야 한다.
12. 시스템은 후보 리밸런싱 초안이 있는 경우 수익 관점과 비용 관점 검토를 실행 판단 전에 포함해야 한다.
13. 최종 판단은 `HOLD`, `DEFER`, `USER_DECISION_REQUIRED`, `REBALANCE` 중 하나로 저장되어야 한다.
14. 시스템은 최종 판단, 후보 리밸런싱 계획, 합의 점수, 충돌 점수, 근거 trace를 저장해야 한다.
15. 사용자는 History 화면에서 과거 판단과 Decision Trace를 확인할 수 있어야 한다.
16. 시스템은 `REBALANCE` 판단에 대해 KIS 현재가를 조회하고 주문 가능 수량을 산출해야 한다.
17. 시스템은 모의투자 주문 실행 전 사용자 확인 게이트를 제공해야 한다.
18. 시스템은 모의투자 주문 요청과 응답 결과를 저장해야 한다.
19. 시스템은 판단 결과에 대해 사용자 피드백과 사후 수익률 평가를 기록할 수 있어야 한다.
20. 시스템은 Buy & Hold, 기계적 리밸런싱, LIBRA 판단 기반 리밸런싱을 비교 검증할 수 있어야 한다.

## 비기능 요구사항

1. KIS API Key와 Secret은 평문 저장하지 않고 암호화하여 보관해야 한다.
2. 실전투자와 모의투자 환경은 명시적으로 분리되어야 한다.
3. 실전 자동매매는 사용자 승인 및 안전장치 검증 전에는 수행하지 않는다.
4. 외부 API 실패 시 판단 실행 전체가 무조건 중단되지 않도록 실패 원인을 기록해야 한다.
5. LLM 응답은 스키마 검증과 정규화를 거쳐 저장해야 한다.
6. 사용자가 이해할 수 있는 한국어 요약과 판단 근거를 제공해야 한다.

## 미완성 또는 고도화 필요 항목

1. KIS 종목 검색 기반 온보딩 UI의 사용성 고도화.
2. Claude 기반 수익/비용 검토 프롬프트와 근거 구조 고도화.
3. 리포트 수집 결과와 Report Agent 판단의 연결 품질 개선.
4. 백테스트 데이터셋과 비교 검증 자동화.
5. 팀원 repo 기능의 활성 repo 반영 내역 문서화.
